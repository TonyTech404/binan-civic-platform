# Bantay Biñan

> See it. Report it. Fix it.

Bantay Biñan is a citizen reporting and civic engagement platform for the City
of Biñan, Laguna. Residents report community issues (potholes, flooding, broken
streetlights, …) with a photo and a map pin — and track the fix from start to
finish. City staff triage, route, and resolve reports from a single dashboard.

**Live:**

| Surface | URL |
| --- | --- |
| Citizen app | https://bantay-binan.buildwithanthony.com · https://bantay-binan.pages.dev |
| API | https://bantay-binan-api.buildwithanthony.com |

## Architecture

Everything runs on **Cloudflare + Supabase**. There is no separate origin
server to operate.

> **Bantay Alerto** (the alert-broadcast app in `apps/alerto`) has its own
> detailed architecture + security overview, with diagram:
> **[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md)**.

```text
  Citizen / Staff browser
            │
            ▼
  React SPA  ──────────────►  Supabase Postgres
  (Cloudflare Pages)          • admin auth (Supabase Auth)
            │                 • RLS-protected reads for staff
            │
            ▼
  Hono API  ──────────────►  Supabase Postgres (service role)
  (Cloudflare Worker)         • anonymous report writes
            │                 • reference numbers, audit log
            ▼
  Cloudflare R2  ─────────►  report & resolution photos
```

**Why this differs from the original spec:** the first draft called for a NestJS
API on Railway. NestJS targets a long-lived Node server and does not deploy
cleanly to Cloudflare's edge runtime. Since the goal was "everything on
Cloudflare," the API is built with **Hono** (the idiomatic Workers framework) —
same role (typed, structured API), but it actually runs on Workers. Citizen
reads/writes flow through the Worker (which holds the Supabase service-role key
and R2 binding); staff reads go directly to Supabase under Row-Level Security.

### Tech stack

| Layer | Choice |
| --- | --- |
| Frontend | React + Vite + TypeScript + Tailwind CSS (shadcn-style UI) |
| Maps | MapLibre GL + OpenStreetMap tiles |
| API | Hono on Cloudflare Workers |
| Database | Supabase Postgres (with RLS) |
| Auth (staff) | Supabase Auth |
| Photo storage | Cloudflare R2 |
| Hosting | Cloudflare Pages (web) + Workers (API) |

```text
bantay-binan/
├── apps/
│   ├── web/          # React citizen app + admin dashboard
│   └── api/          # Hono Cloudflare Worker
├── packages/
│   └── shared/       # categories, statuses, departments, barangays, zod schemas
├── supabase/
│   └── migrations/   # database schema (RLS, triggers, ref-number sequence)
└── package.json      # pnpm workspace
```

`packages/shared` is the single source of truth for the domain model (the 8
categories, 4-step status flow, 5 departments, 24 Biñan barangays, and the Zod
validation schemas used on both the client and the Worker).

## Features

### Citizen portal (no account needed)
- **Submit a report** — category, description, photo, GPS auto-capture + draggable
  map pin, barangay, optional landmark/contact. Targets the < 60-second goal.
- **Track a report** — enter `BB-YYYY-NNNNNN`, see the status timeline, photos,
  and location on a map.
- **Emergency hotlines** — one-tap call links to city services.

### Admin dashboard (Supabase Auth)
- Reports list with **search + filters** (status, category, barangay) and a
  status summary.
- Report detail: **assign a department**, **advance status**, add activity-log
  notes, and **upload a resolution photo**.
- Every mutation is written through the Worker, which records an **audit log**
  and appends to **status history** server-side (citizens can't forge it).

### Status flow

```text
Open → Assigned → In Progress → Resolved
```

## Database

Tables added by this project (additive — the project's pre-existing tables are
untouched):

| Table | Purpose |
| --- | --- |
| `reports` | Main report records (location, photos, status, department) |
| `status_history` | Append-only status timeline (written by triggers) |
| `audit_logs` | Administrative actions, with actor |
| `departments` | Resolution owners (seeded) |
| `report_counters` | Atomic per-year sequence for reference numbers |

Reference numbers are generated by a Postgres trigger
(`next_reference_number()`), so they are gap-free and race-safe. RLS denies
anonymous access to reports; the Worker reads single reports by reference number
using the service role.

## Local development

Prerequisites: Node 20+, pnpm, the Supabase CLI, and Wrangler.

```bash
pnpm install

# Worker API (simulated R2 + local vars from apps/api/.dev.vars) → :8787
pnpm dev:api

# Web app → :5173
pnpm dev:web
```

Environment:
- `apps/web/.env` — `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_API_URL`
- `apps/api/.dev.vars` — `SUPABASE_SERVICE_ROLE_KEY` (gitignored)
- `apps/api/wrangler.toml` — public vars + the R2 binding

See `.env.example` files for the full list.

## Deployment

```bash
# 1. Database — push migrations to the linked Supabase project
pnpm db:push

# 2. API — set the secret once, then deploy the Worker
cd apps/api
echo "$SUPABASE_SERVICE_ROLE_KEY" | wrangler secret put SUPABASE_SERVICE_ROLE_KEY
wrangler deploy

# 3. Web — build with prod env and deploy to Pages
cd apps/web
pnpm deploy          # = vite build + wrangler pages deploy dist
```

Production web env lives in `apps/web/.env.production`. The R2 bucket
(`bantay-binan-photos`) and CORS allowlist (`ALLOWED_ORIGIN` in `wrangler.toml`)
are already configured.

### Custom domains

- **API** `bantay-binan-api.buildwithanthony.com` — live (Worker custom domain,
  configured in `wrangler.toml`).
- **Web** `bantay-binan.buildwithanthony.com` — added to the Pages project but
  **pending a DNS record**. Add a proxied `CNAME`:

  ```text
  bantay-binan  CNAME  bantay-binan.pages.dev   (Proxied)
  ```

  (Cloudflare dashboard → buildwithanthony.com → DNS, or Pages project →
  Custom domains → it offers a one-click "create record".) The app is fully
  usable at `bantay-binan.pages.dev` until then.

### Creating a staff account

Staff are Supabase Auth users. Create one with the Supabase CLI / dashboard, or:

```bash
curl -X POST "$SUPABASE_URL/auth/v1/admin/users" \
  -H "apikey: $SERVICE_ROLE_KEY" -H "Authorization: Bearer $SERVICE_ROLE_KEY" \
  -H "Content-Type: application/json" \
  -d '{"email":"staff@binan.gov.ph","password":"…","email_confirm":true}'
```

Then sign in at `/admin`.
